# Prototype Instructions

Run the local server yourself and open the preview in the browser available to this environment. Do not give the user server-start instructions when you can run it.

When the user says “pousse” or asks to push, publish the verified changes directly to `main` so Render deploys them to production. Do not create a feature branch or pull request unless the user explicitly asks for one.

Before making substantial visual changes, use the Product Design plugin's `get-context` skill when the visual source is unclear or no longer matches the current goal. When the user gives durable prototype-specific design feedback, preferences, or decisions, record them in `AGENTS.md`.

When implementing from a selected generated mock, treat that image as the source of truth for layout, component anatomy, density, spacing, color, typography, visible content, and hierarchy.

## Durable product direction

- The selected visual source is `.design-reference/secret-clubhouse-source.png`.
- Keep the "Secret Clubhouse" indigo, mint, and violet visual language.
- The audience is children ages 6–13 on phones and tablets; touch targets must remain large, friendly, and readable without feeling babyish.
- Keep the child experience deliberately simple: Conversations centers choosing one person and chatting, Clubhouse centers starting one activity, and Mon espace centers changing the avatar or viewing preferences. Child-facing blocks and errors use warm, reassuring language without technical terms or status codes; empty contact states explain the two steps (share the QR/contact ID, then a parent confirms).
- Reduce technical debt incrementally without changing behavior: organize the server behind routes, services, policies, repositories, encryption, notifications, and retention boundaries; organize the interface by authentication, parent, conversations (list, thread, composer, media, calls), Clubhouse, and profile; keep design tokens and shared, domain, and responsive styles separated so fixes remain locally scoped.
- Accounts use parent-managed usernames and QR friend codes, never child phone numbers or public search.
- The parent area requires a real authenticated parent account and focuses on contact approval, safety settings, and high-level activity rather than exposing message contents. Never expose a hard-coded parent PIN or prototype bypass from the child experience.
- An authenticated parent can change their password from the protected parent dashboard only after verifying the current password and confirming a different password of at least eight characters; the password hash is updated in PostgreSQL and the current session remains open.
- A parent can create and manage multiple child profiles (ages 6–13) under one family account. Each child keeps separate contacts and safety settings, and the selected profile becomes the child space shown on the device.
- A parent can permanently delete one of their own child profiles only after an explicit destructive confirmation. Deletion removes the child account and its conversations, messages, media, contact requests, presence, and notification subscriptions; another family can never delete it.
- A family can have one primary parent and invited co-parents with separate e-mail/password accounts managing the same child profiles, contacts, and safety settings without duplicating child data. Invitations are e-mail-bound, single-use, expiring, and explicitly accepted; only the primary parent can invite or remove co-parents and permanently delete a child, and removing a co-parent never deletes family or child data.
- The public entry point is parent-first authentication with separate sign-in and registration forms. Registration opens an empty family and immediately offers first-child creation.
- Before any registration, the public entry point exposes a complete parent privacy policy at `/confidentialite` and a short illustrated child version at `/confidentialite-enfants`. Keep both versions synchronized with the actual Render/PostgreSQL architecture, processors, transfers, retention rules and data-subject rights; never replace them with legal placeholders.
- Registration records the exact versions of the accepted terms, the parental-authority declaration and the privacy notice supplied; privacy information is never presented as consent. Optional notifications require a separate, revocable server-side consent, and for every child profile the server requires the joint agreement of the child and an authenticated family parent before storing or using any push token. Browser or operating-system permissions remain technical permissions and never count as GDPR consent.
- Production exposes no fake account, seeded demo family, demo mode, or notification test control.
- The "Mode calme" safety row opens a per-child schedule editor with separate allowed time windows for messages, audio calls, and video calls.
- The Render API sends its authoritative IANA parental-policy time zone with every child schedule. React evaluates and labels allowed hours in that same time zone, never in the phone’s local time, so travel cannot make the interface disagree with server enforcement.
- Every child profile has a parent-managed private username and password so the child can sign in independently without a phone number; children can never self-register.
- Child usernames are normalized and globally unique login identifiers. Contact IDs shared through QR codes are never accepted by authentication and never contribute to a child username’s identity lockout counter; they remain dedicated to parent-approved contact routing.
- The protected parent inbox clearly separates direct family conversations with the parent’s own children from adult conversations with parents of approved or pending child contacts. It never exposes the children’s conversations with friends.
- A parent can directly open and persist a conversation with any child in their own family without a friend request or approval. Entering the child’s exact contact ID opens that family conversation instead of showing an error.
- Every parent/coparent-to-child conversation is provisioned automatically by the Render service and appears for both participants without requiring the parent to open it first.
- Every family member, including the parent and each child, owns a distinct opaque contact ID. No public contact link is shown; usernames are never used to route discussions, and adding an exact ID only creates a parent-approved request.
- The parent messaging inbox includes an "Ajouter un contact" action. Entering an exact private ID for an external contact creates a pending approval request; entering the ID of one of the parent’s own children opens the family conversation immediately.
- Each child contact modal displays a real scannable QR code containing the app URL and exact private contact ID. Scanning it opens the parent contact-request form prefilled; the parent still confirms sending the request.
- Audio and video calls use WebRTC, are limited to approved contacts, and obey each child’s parent-managed permissions and allowed hours. Camera, microphone, speaker, and hang-up controls must remain obvious.
- Never simulate a call by creating both WebRTC participants in one browser. The caller and callee are separate authenticated accounts; Render/PostgreSQL carries private offer, answer, ICE and call state, while configured STUN/TURN servers establish or relay the media path.
- Parents can configure one neutral automatic reply for messages and audio/video call attempts received outside allowed hours. It must not reveal schedules or personal details, and automatic messages must never trigger another automatic reply.
- Sent messages and media show delivery state consistently in child and parent chats: one check means received and a colored double check means seen.
- Child and parent conversation threads open anchored to the bottom on the latest message and return to the bottom whenever a new sent or received message appears.
- Message delivery and seen states come only from per-recipient PostgreSQL receipts updated by authenticated clients; never advance them with local timers or optimistic fake acknowledgements.
- Conversation lists expose only a decrypted latest-message summary and unread count. Opening a conversation loads bounded cursor-paginated message pages, while periodic synchronization transfers only messages or receipt states newer than the server cursor; never redownload or decrypt the full retained history on a timer, and mark as seen only the message IDs actually loaded for the open conversation.
- Child and parent conversations support recorded voice messages with microphone permission requested on demand, a two-minute limit, preview, cancel, send, playback, and received/seen status.
- The Render API enforces the two-minute voice-message limit before persistence. It inspects the temporary file’s binary format, requires the declared MIME type to match the detected format, derives duration from the media bytes (including block timestamps for headerless MediaRecorder WebM and sample/fragment timing for MediaRecorder MP4), and rejects unverifiable or overlong audio.
- The child Clubhouse is an interactive activity hub with filterable creative challenges and mini-games, private progress, earned stars, daily streaks, and replayable activities.
- Clubhouse progress must label historical stars, active-catalog completion, and the daily streak as three separate server-provided ideas without crowding the header. Activity cards distinguish stars still available from a reward already earned, and all displayed completion, replay, reward and streak values come from Render/PostgreSQL.
- The Clubhouse is a long-term loyalty loop, not a one-pass card list: keep a carefully curated active catalog with 8–12 rotating creative challenges, 2–3 genuinely replayable mini-games, one server-selected daily challenge, and several activities designed for an approved friend or family parent. Preserve historical stars separately from current-catalog discovery; use a compassionate personal streak with a small number of protected missed days instead of a brutal reset. Every activity grants one visible collectible result—such as a badge, Clubhouse color, or decoration—that the child can install, with catalog rotation, daily completion, unlock ownership, and equipped appearance owned by Render/PostgreSQL.
- Clubhouse rewards, unique completed activities, replay counts, daily activity and streaks are persisted per child in Render PostgreSQL. The server owns the reward catalog and awards each activity’s stars only once per child; clients never seed totals or progress locally.
- Mini-games use larger shuffled question banks and serve a fresh three-question set on replay without repeating questions until the current bank has been exhausted.
- Phaser mini-games are bundled inside the React/Capacitor clients, contain no ads or third-party tracking, never open an external browser, and award their activity reward only once per child.
- The Clubhouse includes a private “Multijoueur” hub with server-backed Puissance 4, Morpion, and Bataille navale. Bataille navale uses automatically placed private fleets; the server never exposes an opponent’s unhit ship positions. Two-player games are limited to approved contacts and members of the same family, including children, parents, and co-parents. Adults accept invitations and play from the protected parent area. Invitations and turn-based game state are stored on Render PostgreSQL, synchronized inside the app, resumable, and never expose a public lobby.
- Every eligible child or parent conversation exposes a direct “Jouer” action: tap it, choose Puissance 4, Morpion, or Bataille navale, then explicitly send the private invitation to the locked conversation contact. Real game invitations and their accept/refuse state appear in the conversation without exposing another contact picker.
- The child profile uses one centered 300 px card column: contact ID, protection, notifications, preferences, and sign-out share consistent widths, radii, icon proportions, and spacing on phones and tablets; the indigo background continues without a light gap down to the fixed navigation. It does not expose an “Espace parent” button.
- The child home hero subtitle stays fully visible inside its text column at the 390 px phone layout; it wraps instead of passing beneath the QR friend action.
- Children can send images, photos, and videos only when the parent has enabled media sharing for that child; media remains limited to approved contacts and allowed messaging hours.
- Parent and child conversations display photos, images, and videos as ratio-preserving responsive thumbnails that keep surrounding messages visible. Opening the media and downloading it use separate, clearly labeled controls outside the important image area, with phone- and tablet-appropriate maximum dimensions.
- Media upload requests are capped at 30 MB in total and must declare their length. Multer streams uploads to temporary disk files, the API reads at most one file into memory at a time for PostgreSQL persistence, and temporary files are removed after success, rejection, abort, or failure.
- The protected parent inbox supports parent-to-child and parent-to-parent conversations; adult parent-to-parent threads also support audio/video calls and photo, image, and video attachments.
- Parents in the same family automatically receive one private shared conversation in the protected parent inbox; it is visible to both adults, clearly labeled as a family-parent thread, and removed when a co-parent leaves the family.
- Production persistence uses only a Render Node.js web service and Render PostgreSQL; do not introduce Supabase or another external backend/storage provider. Accounts, messages, and media must use the server API and PostgreSQL as their source of truth.
- Message text, media filenames, media types, media bytes, and WebRTC offer/answer/ICE payloads are encrypted by the Render application before PostgreSQL storage with versioned AES-256-GCM envelopes and explicit key identifiers. Render decrypts content only after participant authorization; this is application-layer encryption, never end-to-end encryption.
- Content-key rotation keeps prior keys explicitly configured until all live rows and eligible backups have moved past them. Startup migrates and validates every retained legacy/old-key message, media and WebRTC signal before listening; a missing key or inconsistent envelope fails closed instead of leaving a partially migrated healthy instance.
- Authentication uses opaque, revocable sessions. In production the web session is delivered only through a Secure, HttpOnly, SameSite=Lax `__Host-` cookie; the native Bearer secret exists only in module memory and disappears when the WebView runtime ends, never entering JavaScript storage. Web requests include cookies but never a Bearer, native requests omit cookies and use the native client header plus Bearer, and the server must match that transport to the stored `client_type`. PostgreSQL stores only each token’s SHA-256 hash, and the default and production session lifetime is 12 hours.
- Production PostgreSQL transport must use either the private internal Render connection or verified TLS with `rejectUnauthorized: true` and an optional trusted CA. Never allow connection-string parameters to disable or bypass this transport policy.
- Online/offline presence is server-backed: authenticated clients heartbeat periodically, PostgreSQL stores last activity, and contacts become offline after the presence timeout.
- New messages and media use standards-based Web Push through the Render service, with subscriptions stored in PostgreSQL. On locked devices the operating system controls notification sound; never promise a custom continuous ringtone from the web app.
- Windows users receive standards-based Web Push through Chrome, Edge, or the installed PWA. Notification settings expose only the Windows-specific activation state; clicking a message notification opens its conversation. In production, stable versioned VAPID key pairs come only from Render secrets; never store a private VAPID key in PostgreSQL. Key rollover must retain the prior pair for its bound subscriptions until clients have re-subscribed to the new public key.
- The production clients are native IPA and APK packages built with Capacitor. Native push uses APNs on iOS and FCM on Android; Render/PostgreSQL remains the only application backend and source of truth.
- Web Push, APNs and FCM payloads expose only opaque routing data and generic visible wording. Never include message content, media filenames, child names or contact names; incoming-call alerts use the neutral label “Contact autorisé”.
- Native notification sounds may use an app-bundled sound through APNs and an Android notification channel. iOS custom alert sounds must remain under 30 seconds; critical-alert behavior is not assumed without Apple entitlement.
- Provide three native notification categories/channels: discreet message alerts, a longer soft ringtone for incoming audio/video calls, and a distinct gentle contact-request sound. Incoming-call ringing stops immediately on accept, decline, cancellation, or timeout. Respect parent quiet-mode schedules and all OS silent/focus settings.
- The intended incoming-call experience uses CallKit on iOS and ConnectionService/Telecom with a high-priority lock-screen call notification on Android, with clear Accept and Decline actions. Do not use or request Apple's critical-alert entitlement for this social app.
- Incoming calls must be answerable or rejectable directly from the native lock-screen call UI. Answering wakes/foregrounds the native client as needed and connects the authorized WebRTC session; the user must never have to manually find and open the app before answering.
- Android and iOS send native call-action tokens only to the configured Secret Clubhouse HTTPS API origin, on the matching port and under `/api/native/calls/`; every action and status request revalidates this allowlist before attaching the token.
- Rejecting an incoming call, from either the lock screen or the app, immediately stops ringing, shows the caller an "Appel refusé" state, and sends one neutral automatic chat message such as "Je ne peux pas répondre pour le moment." The message must not disclose why the call was declined and must never trigger another automatic reply.
- Every child QR modal exposes a real scannable URL carrying only the opaque contact ID; opening that URL prefills the parent contact-request form and still requires explicit parent confirmation.
- The child “Ajouter un ami” action opens identifier entry first, keeps “Mon QR” as a separate tab, and always hands the request to an authenticated parent before submission.
- The child “Mes préférences” area includes an in-app avatar creator with hairstyle, hair color, facial expression, skin tone, and clothing color. The live preview and saved avatar must use the existing indigo, mint, and violet visual language, and production avatar choices persist in Render PostgreSQL.
- Child, family, and adult conversations show a short-lived “is typing” indicator with animated dots. Only the temporary typing state is sent to Render/PostgreSQL; draft text never leaves the sender’s device before sending, and stale indicators expire automatically.
- The child profile must start at the top and scroll when needed so the avatar and “Se déconnecter” action remain fully visible and tappable above the fixed bottom navigation at every supported screen height.
- The protected parent mode is divided into three clear primary spaces: “Accueil” for the selected child’s overview and recent activity, “Gestion” for family profiles, approvals, safety and account settings, and “Conversations” for the protected parent inbox. Keep this three-part navigation fixed in the footer, always visible above the device safe area, and avoid duplicating the same entry points across the parent experience.
- The platform operator has a separate protected administration route showing only aggregate adoption and engagement metrics: total families, active users over 7 and 30 days, 30-day family return, and aggregate session/activity frequency. It never exposes names, contact IDs, relationships, conversations, messages, media, or individual rows; access uses a nominative authenticated parent account, is logged, and excludes the administrator’s own family from every metric.
- The parent experience must feel warm and decision-oriented. “Accueil” answers, in order, whether everything is fine, whether anything needs attention, which child is selected, how protections stand, what recent high-level activity occurred, and what contextually useful action is available now. Pending requests rise to the first card only when they exist; never give an empty zero counter visual prominence. “Gestion” uses progressive disclosure: show essential family and protection status first, then open focused detail views for complex controls such as separate message, audio-call, and video-call schedules. On tablets, “Conversations” uses a persistent two-column master-detail layout with the thread list on the left and the selected conversation on the right.
- On tablets, parent cards use content-driven heights: the compact “Jeux multijoueurs” entry never stretches to match a richer neighbor, and the two “Gestion” columns flow independently without leaving row-sized holes.
- “Mes enfants” grows only when its profile grid needs another row: one and two children stay compact, while four children remain fully visible without horizontal scrolling. On tablet the heading, add action and profile grid share the same row.
- Keep “Installer sur Android” exclusively inside the parent “Compte et application” group with password, data-rights and notification settings. Do not duplicate the installer elsewhere.
- The public parent/child entry screen must remain vertically scrollable in every low-height landscape viewport. Compact the introductory promise in that posture so the authentication card appears immediately and every field, tab, legal link, and submit action remains reachable.
- On landscape tablets, the public entry screen uses the available device width instead of a narrow portrait-sized canvas; the brand, introductory promise, and authentication card remain balanced and visible above the fold without large side gutters or vertically centered overflow.
- Secret Clubhouse is published by Mickael Thorez as a non-professional individual and is currently entirely free, with no subscription, in-app purchase, or advertising. Public legal notices must use the non-professional publisher regime and must not expose the publisher's personal address; the real identification details must instead be kept up to date with Render as host.
- Parental controls are enforced authoritatively by the Render API using PostgreSQL state, never only by React. A paused child cannot use conversations, presence, media, calls, games, push registration, or profile mutations; message, voice, image, video, audio-call, and video-call operations are rechecked server-side against the relevant child schedules, and visual media also requires the child’s media-sharing permission.
- Login attempts are rate-limited persistently in Render PostgreSQL by a keyed hash of the normalized account identifier and by client IP. Five failures for one identity or twenty-five failures from one IP within fifteen minutes cause a fifteen-minute block with a `Retry-After` response; successful login clears only the identity counter.
- Public parent and invited co-parent registrations share a persistent PostgreSQL quota of ten attempts per client IP within fifteen minutes, followed by a fifteen-minute block with `Retry-After`. Invitation existence, status, expiry and bound e-mail are validated before bcrypt, then revalidated under a row lock during acceptance.
- Presence lookup never reveals an arbitrary contact ID. The Render API returns online state only for the authenticated account itself, an approved contact, or another parent/child in the same family.
- Only intentional 4xx `PublicHttpError` messages may pass through the centralized API error boundary; unexpected errors return the generic “Erreur interne.” response. Every HTTP response carries an `X-Request-ID`, centralized error JSON repeats it as `requestId`, and server logs use the same identifier without exposing internal errors to clients.
- Keep automated HTTP application tests for authentication throttling and presence authorization in addition to focused policy unit tests; security-sensitive route changes must preserve those tests. GitHub Actions runs tests, the high-severity npm audit, and the production build on every pull request and push to `main`.
- An external friend request always preserves the exact child or parent profile making the request. A current parent or co-parent of the target child can list, accept, or refuse it; acceptance atomically creates one approved relationship and one private conversation, while refusal grants no messaging or multiplayer access.
- Production applies the documented data-retention policy with explicit PostgreSQL deadlines and a daily Render Cron Job. Inactive family accounts expire after 730 days; text messages after 365 days; media and call metadata after 90 days; WebRTC signals and presence after 24 hours; hashed native-call action tokens after at most 26 hours; push registrations after 180 days; pending contact requests and games after 30 days; security logs after 365 days; Render backups remain limited to 7 days.
- Parent and child profiles expose an age-appropriate “Données et droits” area with immediate readable exports and tracked requests for access, rectification, erasure, restriction, and objection. Requests carry a one-month deadline; restriction blocks ordinary processing at the Render API while leaving rights/export/deletion available. Parent/family/child erasure creates a 30-day PostgreSQL tombstone that must be replayed before any restored backup is put into service.
- A processing-restricted account can call only the explicit method-and-path allowlist needed for account context, rights requests, export, privacy contact, logout, and account/family/child deletion. No wildcard `/api/privacy/*` exception is allowed; notification-consent reads and mutations remain blocked during restriction.
- Secret Clubhouse requires a maintained AIPD because it combines vulnerable children, highly personal communications, regular activity monitoring and mobile/push/WebRTC technologies. Keep `docs/aipd-secret-clubhouse.md` and `server/aipd-register.js` synchronized with every material data, processor, transfer, security or architecture change. Production remains blocked until the named controller formally validates the AIPD and all required evidence is closed; if a high residual risk remains, obtain prior CNIL consultation before the affected processing.
- Keep `docs/registre-sous-traitants-et-transferts.md` synchronized with every provider or hosting change. New Render resources target Frankfurt, while any pre-existing Render service, database, or cron region must be verified and migrated explicitly because Blueprint region changes do not move existing resources. Do not enable a processor-dependent production feature without dated evidence of the applicable contract/DPA, subprocessor review, transfer mechanism and transfer impact assessment; Web Push remains browser-selected and notification payloads stay generic and data-minimized.
- Keep AIPD action `A02` open until a human-reviewed anonymized report is supported by dated human responses from the parent/child consultation. Blank questionnaires, generated answers, refusals, or impossibility records never close `A02` by themselves.
- Keep AIPD action `A04` open until a dated real-world exercise proves named administrators and MFA on Render, GitHub, Cloudflare, Firebase and Apple, least privilege, secret separation, quarterly access review, key rotation, recovery with prior keys and administrator revocation. Unit tests, templates, simulations and unverified dashboard declarations never close `A04`.
- Keep AIPD action `A08` open until a dated, redacted review of the actual Render dashboard proves the live service, PostgreSQL and daily Cron regions, private database transport, required variable names, 12-hour sessions, active and retained content-encryption keys, backup restoration, alert delivery, log retention and green CI for the exact SHA displayed as Live. `render.yaml`, local tests, or CI on another commit never prove the deployed configuration.
